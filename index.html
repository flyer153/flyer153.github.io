也来谈谈团队管理

自己开始带团队也已经有三年多的时间了。经过这一段时间的锻炼，前端团队从最初的4人扩大到整个公司各事业部下的20多人。

从技术转管理最难的就是思维方式的转变，以前你只需要耕耘好你自己的一亩三分地就可以了，至于其他你都可以不用过多的关注。现在作为团队的管理者，你发现你不仅要考虑技术基础架构，考虑整个团队成员的技术水平，还要跨部门的沟通，以及整个团队的任务综合指数。你发现你自己就像一个陀螺一样，每天疲于处理各种各样的需求，特别是创业团队，流程不规范，每天需要处理的事情很多，涉及到跨部门和本部门的沟通也很多，常常会占用你很多的时间。这时候要注意自己时间的管理，比如在某一段时间专门去处理某些零碎的需求，将每天的事情进行分解。及时了解团队里人员的想法和动态。一个团队，里面成员的技术水平一定是有差异的，
评估的一个任务时间，一定要根据团队成员的水平来判断这个人在评估的这个时间段内能不能完成，可能一个任务A来做2天可以完成，交给B来做由于某些原因（技术或者非技术的原因）需要四天完成，这时候leader的角色尤为重要。时间点确认之后，需要告知团队成员确认，在这个时间点上可以完成。leader只需要关注结果，如果在规定的时间内没完成，会有什么损失，是否有补救的办法，投入人力是不是能达到预期，至于过程是开发人员去考虑的事情。
根据团队成员的技术水平，合理分配每个人该负责的工作，把合适的人放在合适的位置，团队leader并非团队技术最牛的那个人，但一定是团队里最有决断权和最能拿下事的人。一定要明确每个人负责的工作，具体能清楚的定位到每个人负责的模块，这样出了问题能第一时间找到问题的责任人，能准确的定位问题。同时事情成功了功劳属于谁也一清二楚。团队是一个整体，出了问题leader要主动的站出来扛起责任和解决问题，不能出了问题就首先去推脱逃避问题和追究是谁的责任，问题已经出现了，最重要的是怎么去解决它，也要让团队的每一个成员明白这个道理。
其实，基层的团队管理者就是要把自己当一个保姆，时刻服务和照顾团队里的每一个人，除非你做到了高层，比如，ceo，vp等级别，那才真正的牵扯到管理学。
如果你待的是创业公司，或者中小型公司，你作为一个部门leader，看到公司技术落后，生产效率低下，你此刻一定是想着要重新搭建一套牛逼的架构来替代。如果你有这种想法，我劝你一定要打住，至少在现阶段打住，因为在这种小公司，生产效率低下的负面影响并不是那么的重要。这种公司几乎都是以运营为导向的，满足业务需要才是最关键的。只要业务发展快，融资节奏跟得上，技术细分领域是否模块化，是否工程化，哪个会在乎呢，页面少几个连接数还是多几个连接数，并不会有立竿见影的影响，高并发上来了，顶不住的话，大不了多几台服务器就是了，但运营和业务的开发一定要优先满足。那就什么都不做了吗？也不是，你要有前瞻意识，技术的模块化以及工程化方面的事情，需要慢慢来，将一部分精力投放到提高团队的生产效率的构建工具开发上面来。

关于前端开发的这点事

工作已经8个年头了，8年，说长不长，说短也不短。8年，整个前端圈子发生了很大的变化，有幸从事这份工作8年，见证了国内前端圈子的变化。

首先是职位名称的变化，从最初的网页美工，网页制作到前端开发，随着职位名称的改变，工作职责也发生了很大的改变.
最初的网页美工、网页制作的工作职责是

  • 熟练运用各类网页制作软件，熟悉Dreamweaver，flash，photoshop等工具；
  • 熟悉HTML/CSS代码，进行布局设计网页；
  • 有优秀的美术功底，网页界面设计能力，视觉设计、色彩有敏锐的观察力及分析能力；

如今我们随便点开一份bat招聘前端开发，可以看到里面的工作职责是

  • 3年以上工作经验，精通基础的前端技术（包括HTML/CSS/JavaScript等），熟悉ES6语法；
  • 精通主流的前端框架，包括React/Vue/Angular等，熟悉RN或Weex者优先；
  • 熟悉网络协议（HTTP/SSL），熟悉常见安全问题和对策；
  • 精通前端构建工具，如 Webpack / FIS，若对前端工程化有自己的实践者优先；
  • 有性能优化，有架构设计能力，有过大规模复杂项目开发背景者优先；
  • 熟悉nodejs 相关知识，并有express/koa等的实践经验，或有其他后端开发经验者优先。

随着现在各互联网公司对前端越来越重视，前端的分工也越来越细。前端相对于其它web技术而言门槛更低，加之现在有很多专业的前端培训机构，所以从事前端的人也越来越多。我是08年实习的时候开始从事前端的职位。那时候还没有这样专业的称呼，都是叫美工，要求的也没有如今这么专业。记得我刚出来的时候什么都不会，切页面也是现学。那时候的美工只需要会点设计，会点html+css就可以找到一份工作。当然工资也不会很高。
如今前端这个职位越来越成熟，市场上找个前端培训机构培训一两个月切出来的页面就比我们当时做一两年切出来的页面还专业了。如果有点程序的基础，js培训半年就可以很熟练的做项目了。

Git 使用

###安装

在Mac上安装Git
第一种，安装homebrew，然后通过homebrew安装Git，具体方法请参考homebrew的文档：http://brew.sh/。
第二种，方法更简单，直接从AppStore安装Xcode，Xcode集成了Git，不过默认没有安装，你需要运行Xcode，选择菜单“Xcode”->“Preferences”，在弹出窗口中找到“Downloads”，选择“Command Line Tools”，点“Install”就可以完成安装了。

###基本操作

####创建版本库

这时候你当前testgit目录下会多了一个.git的目录，这个目录是Git来跟踪管理版本的，不要手动修改这个目录里面的文件，否则会破坏git仓库。

####把文件添加到版本库

用命令 git commit -m “note” 告诉Git，把文件提交到仓库.
现在已经创建一个test.md的文件了，下面可以通过命令git status来查看是否还有文件未提交。
通过命令git status来查看是否还有文件未提交，如果改动test.md,想看下文件到底改了什么内容，可以使用命令：git diff test.md。
知道了对文件做了什么修改后，我们可以放心的提到仓库了，提交修改和提交文件是一样的2步(第一步是git add 第二步是：git commit -m “note”)。

#####版本回退

现在我想使用版本回退操作，我想把当前的版本回退到上一个版本，可以使用如下2种命令，第一是：git reset –hard HEAD^ 那么如果要回退到上上个版本只需把HEAD^ 改成 HEAD^^ 以此类推。那如果要回退到前100个版本的话，可以使用下面的简便命令操作：git reset –hard HEAD~100 即可。

现在我想回退到最新的版本，可以通过版本号回退，使用命令方法如下：
git reset –hard 版本号 ，但是现在的问题假如我已经关掉过一次命令行或者此内容的版本号我并不知道呢？要如何知道增加此内容的版本号呢？可以通过如下命令即可获取到版本号：git reflog

然后可以通过git reset –hard 版本号 来恢复

####工作区与暂存区

工作区：就是你在电脑上看到的目录，比如目录下testgit里的文件(.git隐藏目录版本库除外)。或者以后需要再新建的目录文件等等都属于工作区范畴。
版本库(Repository)：工作区有一个隐藏目录.git,这个不属于工作区，这是版本库。其中版本库里面存了很多东西，其中最重要的就是stage(暂存区)，还有Git为我们自动创建了第一个分支master,以及指向master的一个指针HEAD。
我们前面说过使用Git提交文件到版本库有两步：

第一步：使用git add 把文件添加进去，实际上就是把文件添加到暂存区。
第二步：使用git commit提交更改，实际上就是把暂存区的所有内容提交到当前分支上。

####撤销修改

现在我可以有如下几种方法可以做修改：

第一：如果我知道要删掉那些内容的话，直接手动更改去掉那些需要的文件，然后add添加到暂存区，最后commit掉。
第二：我可以按以前的方法直接恢复到上一个版本。使用 git reset - hard HEAD^
但是现在我不想使用上面的2种方法，我想直接想使用撤销命令该如何操作呢？首先在做撤销之前，我们可以先用 git status 查看下当前的状态。
可以发现，Git会告诉你，git checkout – file 可以丢弃工作区的修改。
意思就是，把file在工作区做的修改全部撤销，这里有2种情况，如下：

第一、file自动修改后，还没有放到暂存区，使用撤销修改就回到和版本库一模一样的状态。
第二、file已经放入暂存区了，接着又作了修改，撤销修改就回到添加暂存区后的状态。

注意：命令git checkout – file 中的-很重要，如果没有-的话，那么命令变成创建分支了。

node.js包管理器

使用 npm 安装包的􏲒􏲣格式为:
npm [install/i] [package_name]

eg: 
npm install express 或 npm i express

本地模式和全局模式

􏳕本地模式: npm install􏲒􏲣

全局模式: npm [install/i] -g [package_name]

eg:
npm install -g supervisor

| 模式 | 可通过require使用 | 注册PATH |
| 本地 | yes | no |
| 全局 | no | yes |

使用场景：
􏳈本地模式：将某个包作为工程运行时的一部分
全局模式：要在􏲒􏲣命令行下使用

创建全局链接： 

npm link(windows不支持)

作用：把全局包当本地包使用(在本地包和全局包之间创建符号链接,因为本地包不支持require)

eg:
    npm install -g express(全局模式)
在工程目录运行:
    npm link express 
    ./node_modules/express -> /usr/local/lib/mode_modules/express
在mode_modules子目录可以看到一个指向安装到全局的包的符号链接

同时也可以将本地包当做全局使用
在包目录(即package.json目录)运行npm link
